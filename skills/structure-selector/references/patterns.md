# Structural patterns

## identity-profile

Use when the main questions are who, what, and how to contact or continue.

Typical sequence:

```text
identity → current role or offer → selected links or proof → contact
```

Avoid adding biography, values, process, and testimonials unless they answer a
real user question.

## chronological-ledger

Use when time explains progression, availability, history, or responsibility.

Record schema:

```text
date or period → organization or item → role or event → optional evidence
```

Prefer stable rows or sections over unique cards for every year.

## writing-index

Use when titles, dates, and topics are the primary retrieval keys.

Record schema:

```text
title → date → topic → optional summary
```

Do not require a generated thumbnail for every item.

## portfolio-index

Use when a reviewer needs to compare projects, roles, years, and results.

Record schema:

```text
project → role → year → scope → verified result or artifact
```

Use images as evidence of work rather than decorative differentiation.

## product-service

Use when a prospective user needs to understand an offer and decide or act.

Typical sequence:

```text
identity and concrete offer
→ operation or use
→ proof
→ price and constraints
→ primary action
→ questions that affect the decision
```

Do not mirror a generic landing-page section checklist.

## collection-catalog

Use for many repeated items sharing a schema.

Record schema:

```text
identifier → name → category → attributes → image or detail
```

High content volume is acceptable when the record grammar remains stable.

## institution-information

Use when current facts, documents, policies, or contact routes are primary.

Typical sequence:

```text
identity → current facts → essential documents → actions → contact
```

Use visible priorities; raw link density alone is not sufficient information
architecture.

## task-workflow

Use for applications where state, action, errors, and completion matter.

Typical sequence:

```text
current state → available action → feedback → next state → completion
```

Visual simplicity must not hide status, validation, or recovery paths.

## justified-hybrid

Use only when the artifact has materially different primary tasks. Define a
clear boundary, such as an editorial index plus a product application. Avoid
mixing several presentation motifs merely for visual variety.
