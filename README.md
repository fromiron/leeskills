# leeskills

`leeskills` is a portable Agent Skills collection for grounding, simplifying,
auditing, and verifying digital interfaces, landing pages, portfolios, product
copy, component libraries, and design systems.

Version: **0.5.0**<br>
Status: **Beta / source-grounded release**  
License: **MIT**

## What this repository does

The skills operationalize a simple principle:

> Use the smallest coherent visual, verbal, and behavioral system that preserves
> the user's task, the artifact's real content, accessibility, and specific
> identity.

"AI slop" is used here as an operational label for generic, unsupported, or
unnecessary output. It is **not** evidence that a design was generated by AI.
The skills must not infer authorship from visual patterns.

## Included skills

| Skill | Job |
|---|---|
| `anti-ai-slop` | Optional orchestrator that chooses and sequences the focused skills |
| `slop-signal-audit` | Finds observable slop signals, scores risk, and prioritizes removals |
| `content-grounding` | Builds an evidence-backed content inventory and blocks fabrication |
| `structure-selector` | Selects one dominant information structure from the user's task |
| `component-contract-audit` | Defines and audits anatomy, states, content, responsive behavior, accessibility, ownership, and design-code parity |
| `visual-entropy-budget` | Constrains visual systems, proposes tokens in HTML, and verifies responsive and nested-radius relationships |
| `specificity-editor` | Rewrites interchangeable marketing copy into supported, concrete copy |
| `motion-necessity-gate` | Keeps motion only when it conveys feedback, state, causality, or context |
| `accessibility-simplicity-guard` | Prevents minimalism from removing accessibility cues or functions |
| `prune-and-verify` | Runs deletion, substitution, reflow, keyboard, motion, and provenance checks |

## Format and portability

Every skill follows the open Agent Skills directory format:

```text
skill-name/
├── SKILL.md
├── references/   # loaded only when needed
├── assets/       # templates and schemas
├── scripts/      # optional deterministic helpers
└── evals/        # trigger and output-quality fixtures
```

Core instructions are vendor-neutral. Optional Python scripts use only the
standard library and perform no network requests.

## Install

Install the collection with the open [`skills` CLI](https://skills.sh/docs/cli):

```bash
npx skills add fromiron/leeskills
```

The CLI discovers all skill directories under `skills/` and prompts for the
target agent and skills. List the available skills or install only one with:

```bash
npx skills add fromiron/leeskills --list
npx skills add fromiron/leeskills --skill anti-ai-slop
```

No npm package from this repository is required; `npx` runs the external
`skills` installer against the GitHub repository.

## Development

Validate the repository:

```bash
python scripts/validate_repo.py
python -m unittest discover -s tests -v
```

Commands use the `python` launcher. Substitute `python3` on Unix-like systems
or `py -3` on Windows when that is the available Python 3.9+ launcher.

For offline installation, a separately maintained clone can use the bundled
Python installer:

```bash
# OpenAI Codex repository scope
python scripts/install.py --client codex --scope repo --mode copy

# Claude Code repository scope
python scripts/install.py --client claude-code --scope repo --mode copy

# Any Agent Skills-compatible client
python scripts/install.py --client generic --target /path/to/skills --mode copy
```

Use `--dry-run` before installation. Existing skills are never overwritten
unless `--force` is supplied.

Trigger fixtures cover English, Korean, and Japanese. After measuring every
query in a target client, evaluate the recorded results with
`scripts/evaluate_trigger_results.py`; see
[`docs/evaluation.md`](docs/evaluation.md). Fixture coverage is not evidence of
an actual trigger rate.

## Recommended workflows

### New interface or landing page

```text
content-grounding
→ structure-selector
→ visual-entropy-budget
→ specificity-editor
→ motion-necessity-gate
→ accessibility-simplicity-guard
→ prune-and-verify
```

### Design system or reusable component

```text
component-contract-audit
→ visual-entropy-budget
→ accessibility-simplicity-guard
→ prune-and-verify
```

### Existing interface audit

```text
slop-signal-audit
→ content-grounding
→ specificity-editor
→ component-contract-audit when reusable components or parity are in scope
→ visual-entropy-budget
→ motion-necessity-gate
→ accessibility-simplicity-guard
→ prune-and-verify
```

### Copy-only review

```text
content-grounding → specificity-editor → prune-and-verify
```

### Quick pass (small artifact or early draft)

```text
slop-signal-audit in quick-pass mode
(single pass, no scores or verdict, up to five changes, skipped checks disclosed)
```

## Evidence model

All findings should be labeled as one of:

- **Observed** — directly visible in supplied copy, screenshots, markup, code,
  design files, or tokens.
- **Measured** — produced by a deterministic test or calculation.
- **Inferred** — reasoned from available evidence and explicitly identified as
  inference.
- **Unknown** — cannot be verified from the supplied material.

Do not upgrade an inference to a fact. Do not invent customer names, metrics,
quotes, awards, product capabilities, project outcomes, authorship, or
accessibility compliance.

## What the score means

`slop-signal-audit` uses a 100-point quality rubric and derives a slop-risk
score from it. This is a prioritization aid, not a scientific measurement and
not a detector of AI authorship.

An `unknown` category may receive at most half of its category maximum. Only
observed or measured evidence may create a project hard failure.

Project hard failures block a passing verdict regardless of score. Examples
include fabricated evidence, fake product UI presented as real, removed
keyboard focus, and scroll hijacking without an equivalent user-controlled
mode.

## Design foundations

The rules draw from:

- the curation criteria stated by Dead Simple Sites;
- a maintainer-reviewed minimal-site corpus, with observations kept separate
  from interpretation;
- the Codeit Design System's foundation and component documentation patterns;
- the open Agent Skills specification and its progressive-disclosure model;
- W3C WCAG 2.2 guidance for contrast, reflow, keyboard focus, target size,
  semantics, and interaction motion;
- task-oriented content-design guidance.

See [`docs/source-notes.md`](docs/source-notes.md) for provenance,
[`docs/corpus-study.md`](docs/corpus-study.md) for the corpus evidence boundary
and repeatable sampling protocol, and
[`docs/design-foundations.md`](docs/design-foundations.md) for normalized
principles.

## Limits

- A screenshot cannot establish full accessibility conformance or component
  runtime behavior.
- A static audit cannot prove user comprehension or conversion impact.
- Generic phrase and pattern linting produces false positives; context and
  evidence decide.
- The current minimal-site contrasts are source-grounded maintainer heuristics,
  not a completed systematic corpus study.
- Visual budgets are defaults, not universal aesthetic laws.
- Client-specific trigger behavior is nondeterministic and must be evaluated in
  the target agent.

## Versioning

Use semantic versioning:

- patch: clearer wording, fixtures, or non-behavioral corrections;
- minor: new checks, optional fields, or backward-compatible skill behavior;
- major: changed output contracts, scoring, or required workflow order.
